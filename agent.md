# SmartCare Agent 工作说明

## 1. 角色定位

- 你是 SmartCare 项目的工程协作 Agent。
- 目标是快速、准确、可追踪地完成需求，优先交付可运行结果。

## 2. 工作原则

- 先理解需求，再改代码，最后自检。
- 优先小步提交，避免一次性大改。
- 不改与当前任务无关的文件。
- 涉及接口、数据库、配置变更时，同步更新文档。

## 3. 目录协作约定

- `frontend/`: 前端应用代码
- `backend/`: 后端服务代码
- `ai-service/`: AI 服务相关代码
- `database/`: 数据库脚本与结构
- `docs/`: 项目文档

## 4. 提交流程（必须）

1. 新建任务分支（建议）：`feat/<module>-<desc>` 或 `fix/<module>-<desc>`
2. 完成最小可交付改动
3. 本地自检（编译/运行/关键测试）
4. 使用规范提交信息
5. 发起合并请求并说明影响范围

## 5. 提交信息规范（Conventional Commits）

格式：

```text
<type>(<scope>): <subject>
```

常用 `type`：

- `feat` 新功能
- `fix` 缺陷修复
- `refactor` 重构
- `perf` 性能优化
- `docs` 文档变更
- `test` 测试变更
- `build` 构建或依赖变更
- `ci` CI/CD 变更
- `chore` 杂项任务
- `revert` 回滚

示例：

```text
feat(backend): 新增患者随访接口
fix(frontend): 修复提醒时区显示错误
docs(api): 更新鉴权错误码说明
```

## 6. 变更质量要求

- 代码可读：命名清晰、逻辑分层。
- 变更可测：至少覆盖本次改动路径。
- 结果可回滚：避免耦合过高的大提交。
- 风险可说明：已知限制和影响范围要写清楚。

## 7. 安全与合规

- 禁止提交密钥、token、密码、隐私数据。
- 配置项优先走环境变量。
- 涉及患者数据处理时，默认最小化数据暴露。

## 8. 文档同步

以下场景必须更新 `docs/`：

- 新增或修改 API
- 数据库结构变化
- 核心业务流程变化
- 部署或运行方式变化
- 项目结构变化
